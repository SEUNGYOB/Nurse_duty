# Claude OCR Findings

## 목적

Claude Vision 기반 근무표 OCR에서 무엇을 시도했고, 무엇이 실패했고, 다음 작업 시 무엇을 먼저 확인해야 하는지 기록한다.

## 현재 결론

- 회전 보정 자체는 동작한다.
- 현재 병목은 `180도` 회전이 아니라 `휘어진 표에서 행/날짜 매핑`이다.
- Claude Vision 단일 호출은 전체 표를 전역적으로 해석하는 경향이 강해, 한 행의 날짜 정렬이 틀어질 수 있다.
- 프롬프트만 강화해서는 동일 오답 패턴을 없애지 못했다.
- 세로 해석 경로는 현재까지 두 번 모두 실패했고, 보조 경로로도 효과가 없었다.
- 지금 가장 유망한 방향은 `가로 전체표 1회 판독 + 문제 행만 재판독`이다.

## 확인된 사실

### 1. 회전 보정은 문제의 핵심이 아니다

- `0도` 원본과 `180도` 회전본 모두 같은 정확도와 같은 오답 패턴이 나왔다.
- 자동 회전 결과:
  - 원본은 `rotationApplied = 0`
  - 180도 회전본은 `rotationApplied = 180`
- 따라서 회전 보정은 작동하지만, OCR 오차의 원인은 별개다.

### 2. 현재 Claude 단일 호출 결과는 97.08%로 반복 재현된다

- 샘플 기준 `466 / 480`
- `exactRows = 15`
- 오답은 모두 `장혜진` 행 하나에 몰렸다.

### 3. 오답은 한 행의 날짜 매핑 이상처럼 보인다

- 위 행 `이현진`과 아래 행 `이영미`는 100% 정답
- `장혜진` 행만 `D / E / OFF`가 반복적으로 어긋남
- 오답은 단순 전체 1칸 shift가 아니라, 일부 날짜 구간에서만 날짜 대응이 흐트러진 패턴이다.

## 시도한 방법과 결과

### 1. EXIF + 자동 회전 보정

무엇을 했나
- EXIF 방향 보정 추가
- `90 / 180 / 270도` 회전 후보 비교
- `180도`는 좌측 라벨 열과 우측 듀티 그리드 비대칭으로 판별

결과
- 원본/180도 회전본 모두 정상 방향으로 복구
- 속도도 큰 문제 없음

왜 핵심 해결이 아니었나
- 회전 후에도 같은 행에서 같은 오답 패턴이 유지됨

### 2. 프롬프트 강화: 연속성은 위치 연속성이라고 명시

무엇을 했나
- 한 행을 왼쪽에서 오른쪽으로 연속된 공간 시퀀스로 읽으라고 지시
- 인접 셀은 값 추론이 아니라 경계 추론에만 사용하라고 명시
- 위아래 행 복사 금지, 좌우 shift 금지 지시 추가

결과
- 정확도 변화 없음
- `장혜진` 행 오답 패턴 그대로 유지

왜 실패했나
- 프롬프트는 실제 반영됐지만, 전체 표 단일 호출의 전역 해석 방식 자체가 바뀌지 않음

### 3. 프롬프트 반영 여부 확인용 sentinel 테스트

무엇을 했나
- 테스트용으로 `장해진 행은 30칸 모두 null`을 반환하라고 강한 문구 삽입

결과
- 실제로 `장혜진` 행이 모두 `null` 처리됨
- 동시에 다른 여러 행 정확도도 크게 무너짐

무엇을 배웠나
- 프롬프트 반영은 된다
- 캐시 문제는 아니다
- Claude는 표를 행별 독립 문제로 푸는 것이 아니라 전체 표를 전역적으로 함께 해석한다

### 4. 코드 기반 row strip 분할

무엇을 했나
- `detect_schedule_line_bounds` 결과로 16개 row strip 생성
- overlay와 contact sheet로 시각 검증

결과
- 상단 몇 줄은 그럴듯하지만, 중간 이후는 위아래 행을 함께 먹거나 빈 strip가 나옴
- 하단부는 특히 무너짐

왜 실패했나
- 사람 눈에는 괜찮은 곡률도, 직선 기반 규칙 분할은 잘 못 따라감
- `한 행 = 하나의 직선 y 범위` 가정이 약함

### 5. 장혜진 행 crop 검수

무엇을 했나
- 고정 박스 기준 crop
- grid line 기반 동적 crop

결과
- 고정 crop은 위 줄 일부를 먹는 듯 보였음
- 동적 crop도 한 행만 깨끗하게 분리되지는 않음

무엇을 배웠나
- 단순 crop만으로는 원인을 확정하기 어렵다
- 문제는 문자 인식보다 `행/열 경계 정렬`일 가능성이 높다

### 6. 동적 셀 후보 `U / M / D` 실험

무엇을 했나
- `장해진` 행의 각 날짜 셀에 대해 위/중간/아래 후보 3개 생성

결과
- 후보 세트는 생성 가능
- 하지만 후보 간 차이가 너무 작고, 세로 이동만으로는 후반부 문제를 해결하기 어려워 보임

왜 부족했나
- 실제 문제는 세로 오프셋만이 아니라 좌우 경계와 폭도 함께 흔들릴 수 있음
- `27~30일` 같은 구간은 세로 이동만으로는 충분한 후보 다양성을 만들지 못함

### 7. 세로 열 재조립 1차: 원본 열 strip 그대로 읽기

무엇을 했나
- 하루 단위 세로 열을 길게 crop해서 Claude가 위에서 아래로 16개 값을 읽게 함
- 30일 결과를 다시 행 기준으로 재조립함

결과
- 최종 정확도는 약 `35.83%`
- 오답에 `12`, `21`, `↑`, `~` 같은 숫자/기호가 섞여 나옴

왜 실패했나
- 열 strip 안에 헤더/주석/격자 잔상이 함께 들어가 있었다
- 이미지가 너무 좁고 길어서 Claude가 실제 16개 근무칸보다 헤더 숫자와 기호를 값으로 읽었다
- 즉 전략 자체보다 `입력 포맷`이 나빴다

### 8. 세로 열 재조립 2차: 16개 셀을 카드처럼 다시 세워 붙이기

무엇을 했나
- 각 day column에 대해, 코드가 검출한 16개 셀 box를 개별 crop
- 각 셀을 같은 크기 카드로 정규화
- 왼쪽에 row 번호와 이름을 붙이고, 오른쪽에 셀 카드 1개만 보이도록 `guided day board` 생성
- 즉 `좁고 긴 열 strip` 대신 `16개 독립 카드의 세로 리스트`로 바꿈

결과
- 최종 정확도는 약 `30.0%`
- `nullCells = 45`
- 하단 `김성남`, `김우진` 행은 여러 날짜가 거의 통째로 `null`
- 상단/중단부는 `S / OFF / N` 쪽으로 과하게 쏠리는 오인식이 반복됨

왜 실패했나
- 헤더 숫자/기호 누수는 줄였지만, 카드화하면서 셀 문맥이 지나치게 단순화됐다
- Claude가 실제 표를 읽기보다 비슷한 카드 묶음에서 패턴을 과잉 일반화한 것으로 보인다
- 하단부는 원본 cell crop 품질이 약해서 카드화 후 정보가 더 줄었다

무엇을 배웠나
- 세로 방식의 문제는 단순 헤더 누수가 아니라, Claude가 세로 재조립 입력 자체를 덜 자연스럽게 해석한다는 점이다
- `raw column strip`이든 `guided day board`든 세로 단독 파이프라인은 현재 메인 경로로 쓰기 어렵다

### 9. 가로/세로 교차검증 아이디어

무엇을 했나
- 가로 전체표 결과와 세로 재조립 결과를 함께 비교할 수 있는 `benchmark_crosscheck.py` 경로를 만들었다
- 목적은 어떤 셀은 가로가, 어떤 셀은 세로가 더 잘 읽는지 확인하는 것이었다

결과
- 실제로는 세로 해석이 너무 많이 틀려서 `cross-check`가 아니라 `noise source`가 되었다
- 사용자가 확인한 결론도 동일했다: `수직 해석이 문제`

왜 실패했나
- 세로 결과가 너무 불안정해서, 교차검증에 쓸 정도의 보조 신호가 되지 못했다
- 따라서 현재 단계에서 가로/세로 fusion은 의미가 없다

### 10. 문제 행만 재판독하는 보조 루프 추가

무엇을 했나
- 전체표 가로 판독 결과는 유지
- 특정 rowIndex만 다시 읽는 보조 경로를 추가했다
- 재판독 시에는
  - 전체 표에서 대상 행을 강조한 이미지
  - 날짜 헤더와 대상 행이 함께 보이는 focused crop
  - 이 두 이미지를 Claude에 같이 보내도록 구현했다
- 서버에서는 `claudeRefineRows` 파라미터로 재판독 대상을 받을 수 있게 했다
- 벤치마크 스크립트에도 `--claude-refine-rows` 옵션을 추가했다

왜 이 방향이 유망한가
- 현재 최고 성능은 여전히 가로 전체표 단일 호출 `97.08%`
- 오답도 `장혜진` 한 행에 몰려 있으므로, 전체를 갈아엎지 않고 문제 행만 다시 읽는 편이 훨씬 합리적이다
- 세로 전략처럼 완전히 다른 구조를 만들지 않고, 기존 강점을 유지할 수 있다

아직 확인이 필요한 것
- `rowIndex 4` 재판독이 실제로 `장혜진` 행 정확도를 개선하는지
- 전체 정확도가 `97.08%`를 넘는지
- 재판독 추가 시간 비용이 어느 정도인지

## 왜 첫 행은 잘 읽고, 장혜진 행은 틀릴 수 있나

- 첫 행은 헤더 바로 아래라 날짜 정렬 기준점에 더 가깝다.
- 첫 행은 위쪽이 헤더라서 행 경계가 더 뚜렷하다.
- `장혜진` 행은 중간 행이고, `D / E / OFF` 반복 패턴이 많아 날짜 매핑이 조금만 흔들려도 그럴듯한 오답이 나오기 쉽다.
- 즉 곡률 크기 자체보다 `행 위치 + 값 패턴의 모호성` 영향이 크다.

## 지금 단계에서의 판단

- Claude Vision만으로 이 문제를 안정적으로 푸는 건 어렵다.
- 그래도 바로 다른 OCR로 갈아타기 전에, Claude의 역할을 쪼개는 실험은 해볼 가치가 있다.
- 세로 해석은 현재까지 실패로 보는 것이 맞다.
- 다음 실험 우선순위는 세로 개선이 아니라 `가로 유지 + 문제 행 재판독`이다.

## 최종 결론 (2026-05-27 벤치마크 확정)

벤치마크 결과 (`주희_duty_june.jpeg`, 480셀):

| 방식 | 정확도 | 시간 | 호출 | 비용 |
|------|--------|------|------|------|
| 베이스라인 (보조선 없음) | 97.1% | 32.7s | 1 | $0.036 |
| **파란 보조선만 (채택)** | **100%** | 32.4s | 1 | $0.041 |
| 재판독만 row4 | 100% | 37.3s | 2 | $0.045 |
| 파란 보조선 + 재판독 | 100% | 38.0s | 2 | $0.050 |

**채택: 파란 보조선 (`use_row_guides=True`)** — 시간/비용 동등, 호출 1회, 100% 달성.

재판독(`refine_row_indices`)은 보조선 없이도 100% 가능하나, 속도·비용 모두 불리. 보조선과 조합해도 정확도 차이 없음 → 현재 미사용.

## 미래에 다시 검토할 옵션

- **자동 재판독 트리거**: 1차 결과에서 null 비율이 높은 행을 자동 감지해 `refine_row_indices`로 재판독. null이 특정 임계값(예: 5개 이상) 초과 시만 2차 호출.
- **PaddleOCR-VL-1.5**: Claude 비용이 문제가 될 규모로 성장하거나, 다른 병원 양식에서 정확도가 떨어질 경우 검토.
- **다른 병원 양식 테스트**: 현재 샘플 1장 기준이므로 표 레이아웃이 다른 사진에서 보조선이 여전히 효과적인지 확인 필요.
